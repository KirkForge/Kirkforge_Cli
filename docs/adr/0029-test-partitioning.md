# ADR-0029: Test partitioning — fast / full / coverage suites

- **Status:** Accepted
- **Date:** 2026-07-21
- **Related:** [ADR-0016](./0016-test-strategy.md) (test categories),
  [test-doctor idea](../ideas/test-doctor.md)

## Context

KirkForge-Cli has 2772+ tests. Local runtime is ~98s, but a single CI
job takes 7-8 minutes because every job re-runs the full workspace
suite:

| Job | Command | Wall time |
|-----|---------|-----------|
| `quality` | `cargo test --workspace` | ~7 min |
| `coverage` | `cargo tarpaulin --workspace` | ~4 min |
| `windows` | `cargo test --workspace` | ~7 min (continue-on-error) |
| `integration` | `cargo test --test integration_test -- --include-ignored` | ~3 min |

The wall-clock budget of a PR is dominated by **re-running the same
suite under different harnesses**, not by the cost of any individual
test. The integration tests in `tests/integration_test.rs` spawn a
live Ollama model and take ~38s for 14 tests (2700ms/test), yet they
are re-instrumented by tarpaulin even though the coverage gate
(see `.github/workflows/ci.yml` `Enforce coverage gate` step) only
enforces thresholds on `src/session`, `src/tools`, `src/adapters` —
all unit-test-covered code.

The same cliff hit PicoSentry's Python suite; the solution there was
`scripts/test_doctor.py`, a diagnostic that runs the CI checks in
parallel and reports per-area timings. The Rust equivalent needs the
same diagnostic plus **suite partitioning** so the coverage job does
not pay for integration tests it does not measure.

## Decision

Partition the test suite into three CI manifests, generated by the
`kirkforge-testdoctor` prototype (see
[`docs/ideas/test-doctor.md`](../ideas/test-doctor.md)):

1. **`fast-suite`** — `cargo test --lib` (lib targets only). Target:
   < 60s. Run on **every PR**.
2. **`full-suite`** — `cargo test --workspace` (all non-ignored tests).
   Run on **merge to `main` / `dev` / `merge` / `phase/**`**.
3. **`coverage-suite`** — `cargo tarpaulin --lib` (unit tests only).
   Run in the **coverage job** on every PR and push.

The partition is derived from a profile (`test-profile.json`) that
records per-binary timings. Each binary is classified by average
milliseconds per test:

| Class | Per-test avg | Binary total | Goes into |
|-------|-------------|--------------|-----------|
| `fast` | ≤ 100ms | < 60s | fast, full, coverage |
| `medium` | 100-500ms | < 60s | fast, full, coverage |
| `slow` | > 500ms | any | full only (+ `--ignored` job) |
| `ignored` | n/a | n/a | dedicated `--ignored` job |

The KirkForge-Cli lib suite (43s / 1287 tests = 33ms/test) is
`fast`; the integration suite (38s / 14 tests = 2700ms/test) is
`slow`. The coverage gate thresholds are unchanged — they already
target unit-test-covered code.

### CI workflow changes (`.github/workflows/ci.yml`)

1. **`quality` job** — install `cargo-nextest` via
   `taiki-e/install-action@v2`; on `pull_request` run
   `cargo nextest run --lib --workspace` (fast suite), on `push` run
   `cargo nextest run --workspace` (full suite). Fall back to
   `cargo test` if nextest fails to install.
2. **`coverage` job** — change `cargo tarpaulin --workspace` to
   `cargo tarpaulin --lib`. The coverage gate is unchanged.
3. **`.tarpaulin.toml`** — fix `timeout = "120"` → `timeout = "120s"`
   (tarpaulin expects a duration string; the bare number was silently
   rejected and replaced with the default 60s).

The `windows` and `integration` jobs are unchanged. The `windows` job
keeps `cargo test --workspace` with `continue-on-error: true` (parity
is a work in progress per ADR-025). The `integration` job already
runs only `--test integration_test -- --include-ignored` and
`--test smoke_test` — it does not run the full suite.

### `kirkforge-testdoctor` prototype

A standalone crate at `crates/kirkforge-testdoctor/` (not a workspace
member yet — prototype only) provides four subcommands:

- `profile` — runs `cargo test --workspace --no-fail-fast` and parses
  the per-binary `test result:` lines into `test-profile.json`.
- `classify` — reads the profile and classifies each binary.
- `partition` — writes `fast-suite.json`, `full-suite.json`,
  `coverage-suite.json`.
- `suggest` — prints fix suggestions for slow binaries (mark
  `#[ignore]`, use `tokio::time::pause`, mock the subprocess, move to
  `tests/`).

On stable Rust 1.88 the nightly-only `--format json` test output is
not available, so the doctor parses the standard text output. This
gives per-binary timings (not per-test) on stable; per-test timings
require nightly or `cargo-nextest --format json` (nextest's JSON is
stable).

## Consequences

Negative first:

- **PRs no longer run integration tests.** A regression in
  `tests/integration_test.rs` that is not caught by the lib suite
  will land on `dev` and only fail the `integration` job (which
  already runs on PRs) or the next `push` to `main`. Mitigation: the
  `integration` job is unchanged and still gates PRs.
- **Coverage no longer measures integration tests.** This is
  acceptable because the coverage gate never enforced thresholds on
  integration-test code — only on `src/session`, `src/tools`,
  `src/adapters`.
- **The doctor is a prototype.** It is not a workspace member; CI
  does not invoke it. The partition is currently encoded directly in
  the workflow `if`/`else`. Promoting the doctor to a workspace
  member and driving CI from the generated manifests is a follow-up.

Positive:

- **Coverage job: ~4 min → ~2 min.** Tarpaulin instruments the lib
  suite only, skipping the 38s integration suite and the
  subprocess-spawning tests that tarpaulin's tracing engine handles
  poorly.
- **Quality job on PRs: ~7 min → ~3 min.** The lib suite is ~43s
  locally; nextest's better parallelism plus skipping integration
  tests brings the PR test step to ~60-90s (plus ~40s build, ~30s
  clippy, ~30s release build).
- **Quality job on push: unchanged coverage, faster via nextest.**
  The full suite still runs on merge to `main`, so integration
  regressions are caught before release.
- **`cargo-nextest` is 2-3x faster than `cargo test`** on top of the
  partitioning win, because nextest parallelises test binaries and
  individual tests more aggressively and retries failures in
  isolation.
- **The `.tarpaulin.toml` `timeout` fix** removes a silent
  misconfiguration where the 120s intent was replaced by the 60s
  default.

## Expected speedup

| Job | Before | After | Lever |
|-----|--------|-------|-------|
| `quality` (PR) | ~7 min | ~3 min | `--lib` + nextest |
| `quality` (push) | ~7 min | ~5 min | nextest only |
| `coverage` | ~4 min | ~2 min | `--lib` + timeout fix |
| `integration` | ~3 min | ~3 min | unchanged |
| `windows` | ~7 min | ~7 min | unchanged |
| `fmt` / `changelog` / `audit` / `node-sdk` | < 1 min each | unchanged | n/a |

**PR wall time (critical path):** ~7 min → ~3 min (quality job is the
long pole; coverage runs in parallel).

## Implementation notes

The workflow uses `github.event_name` to distinguish PR from push:

```yaml
- name: Run tests (fast suite on PR, full suite on push)
  run: |
    if [ "${{ github.event_name }}" = "pull_request" ]; then
      cargo nextest run --locked --lib --no-fail-fast --workspace || cargo test --locked --lib --no-fail-fast
    else
      cargo nextest run --locked --workspace --no-fail-fast || cargo test --locked --workspace --no-fail-fast
    fi
```

The `|| cargo test ...` fallback handles the case where nextest fails
to install on the runner. `--locked` preserves the original
lockfile-fidelity guarantee.

The `coverage` job change is a single flag:

```diff
-        run: cargo tarpaulin --out Xml --locked --workspace --timeout 120
+        run: cargo tarpaulin --out Xml --locked --lib --timeout 120
```

The `.tarpaulin.toml` fix:

```diff
-timeout = "120"
+timeout = "120s"
```

The `kirkforge-testdoctor` crate is built standalone
(`cd crates/kirkforge-testdoctor && cargo build`) and is not listed
in the root `[workspace] members`. It depends only on `clap`,
`serde`, `serde_json`, and `anyhow` — all already in the workspace
dependency tree.
