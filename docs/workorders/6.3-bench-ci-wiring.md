# Workorder 6.3 — Wire Bench CI Job Properly

## Status: Planned

## Problem

The `bench` CI job (`.github/workflows/ci.yml:186-235`) has four issues:

1. **Skipped when upstream fails**: `needs: [fmt, quality]` with no `if: always()`
   means the bench job is skipped whenever fmt or quality fail — the common case
   during development. You never get bench data on a red PR.
2. **Non-gating**: `|| true` means it never fails CI, so pass-rate regressions
   are invisible to branch protection.
3. **No PR comment**: ADR-038 claims the summary is "posted as a PR comment" but
   the workflow only `cat`s the markdown to the log.
4. **No path filter**: runs on every push/PR regardless of whether agent code
   changed, burning CI minutes.

## Root cause

The bench CI job was wired as a proof-of-concept during the P0 bench build and
never hardened.

## Files to touch

- `.github/workflows/ci.yml` — rework the `bench` job:
  - Add `if: always() && (needs.fmt.result == 'success' || needs.fmt.result == 'skipped')`
    so it runs even when quality fails (but not when fmt fails — a broken build
    can't bench).
  - Add a `paths` filter or a `paths-ignore` so the job only runs when
    `src/session/**`, `src/adapters/**`, `src/tools/**`,
    `crates/kirkforge-bench/**`, or `benches/tasks/**` change.
  - Add a baseline-fetch step: download the last `main` bench report from a
    `bench-baseline` artifact (or a committed JSON under `benches/baselines/`).
  - Replace `|| true` with a real (but lenient) gate: fail if
    `current.success_rate < baseline.success_rate - 10` (10-point tolerance for
    LLM noise). Keep informational-only as a fallback for the first few runs
    until baselines stabilize.
  - Add a PR-comment step using `gh pr comment` or `actions/github-script` to
    post the delta markdown (from Workorder 6.2).
- `.github/workflows/bench-baseline.yml` (new) — a scheduled workflow that runs
  `kirkforge bench` on `main` nightly and on merge to main, uploading the report
  as a `bench-baseline` artifact.

## Approach

1. Split the bench job into two: `bench-run` (runs the tasks, uploads report as
   artifact) and `bench-report` (downloads baseline, compares, posts PR
   comment). This lets `bench-run` be fast and `bench-report` be optional.
2. The baseline workflow runs on schedule (cron: `0 4 * * *` — once nightly) and
   on push to `main`. It runs the same `kirkforge bench` command and uploads the
   JSON report.
3. The PR comment step uses `gh pr comment ${{ github.event.pull_request.number }}
   --body-file delta.md` (requires `permissions: pull-requests: write`).

## Gate

CI validation only — push to a PR branch and confirm:
- The bench job runs even when quality fails (temporarily break a clippy lint
  in a separate commit to test).
- The PR comment appears with the delta table.
- The path filter skips the bench job on a docs-only change.

## Done condition

- Bench job runs independently of quality.
- Path filter prevents unnecessary runs.
- PR comment posts the delta markdown.
- Baseline workflow produces a nightly `main` report.
- Depends on: Workorder 6.2 (delta comparison) for the PR comment content.
