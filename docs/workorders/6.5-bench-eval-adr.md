# Workorder 6.5 — ADR for Continuous Eval Pipeline

## Status: Planned

## Problem

ADR-038 (`docs/adr/038-task-benchmark-harness.md`) describes the benchmark
harness design intent, including claims like "every PR that touches agent code
gets a benchmark delta" and "posts the markdown summary as a PR comment." These
claims are aspirational — the delta and PR-comment infrastructure did not exist
when the ADR was written. After Workorders 6.1-6.4 deliver the actual pipeline,
ADR-038 must be updated to reflect reality, and a new ADR should pin the
continuous evaluation pipeline design.

## Root cause

ADR-038 was written ahead of implementation. The implementation lagged. The ADR
overclaims.

## Files to touch

- `docs/adr/038-task-benchmark-harness.md` — update:
  - Status: keep "Accepted" but add an "Implementation notes" section that
    honestly describes what shipped (single-run harness, 10 tasks) vs what was
    added in 6.1-6.4 (realism fix, delta comparison, CI wiring, verify-only).
  - Fix the overclaiming language ("posts the markdown summary as a PR comment"
    → describe the actual `gh pr comment` step).
- `docs/adr/0045-continuous-eval-pipeline.md` (new) — pin the continuous
  evaluation pipeline design:
  - Nightly baseline on `main`
  - Per-PR run with path filter
  - Delta comparison with 10-point tolerance
  - PR comment with delta table
  - `verify-only` as a fast deterministic smoke
  - Non-gating initially, gating once baselines stabilize
- `docs/adr/README.md` — add ADR-045 to the index table.
- `crates/plugin3-core/tests/adr_xref_drift.rs` — no change needed (test is
  generic), but run it to confirm the new ADR is picked up.
