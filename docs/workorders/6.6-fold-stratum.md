# Workorder 6.6 — Fold Stratum into Core

## Status: Planned

## Problem

Stratum (`crates/kirkstratum-core`) is a context-compression library that is
not linked into the main `kirkforge` binary. It is invoked via shell scripts
(`plugins/stratum/tools/*.sh`) that exec the standalone `stratum` binary. This
adds subprocess overhead, requires the binary to be on PATH or in
`target/`, and prevents the compression logic from accessing full session
context.

## Root cause

Stratum was originally a standalone CLI plugin. It was added to the workspace
for shared builds but never linked as a library dependency of the root crate.

## Files to touch

- `Cargo.toml` (root) — add `kirkstratum-core = { path = "crates/kirkstratum-core", optional = true }` and feature `stratum = ["dep:kirkstratum-core"]`; add `stratum` to `default`.
- `src/shared/config/tools.rs` — add a `stratum_mode` config field (off/lite/full/ultra) defaulting to `full`; the existing `enabled_plugins` toggle for "stratum" continues to work as the user-facing on/off.
- `src/session/plugin_tools/loader.rs` — when the `stratum` feature is enabled, register Stratum's 5 tools as direct Rust calls instead of `PluginToolWrapper` shell-outs. When disabled, fall back to the shell-plugin path (or skip entirely).
- `src/session/hooks.rs` — when the `stratum` feature is enabled, register the 2 hooks (`session-start`, `pre-tool-bash`) as in-process Rust handlers instead of shell scripts.
- `src/session/stratum.rs` (new) — a thin module that wraps `kirkstratum_core::CompressionPipeline` and exposes 5 functions matching the tool names: `run`, `apply`, `rules`, `mode`, `config_validate`. Each returns JSON matching the shell tool's output format.
- `src/session/plugin_tools/tests.rs` — update the `default_plugin_sources_are_present_and_loadable` test if the stratum plugin is no longer a shell plugin.
- `docs/adr/0046-stratum-fold-in.md` (new) — pin the decision to fold Stratum into core behind a feature flag, with the shell-plugin path as fallback.

## Approach

1. Add `kirkstratum-core` as an optional dep + feature in root `Cargo.toml`.
2. Create `src/session/stratum.rs` with 5 functions that call
   `kirkstratum_core` directly and return JSON strings.
3. In the plugin loader, when the `stratum` feature is active, register these
   functions as `Tool` implementations (a `StratumToolWrapper` implementing
   `Tool`) instead of `PluginToolWrapper`.
4. Register the 2 hooks as in-process handlers (Rust closures) instead of shell
   scripts.
5. When the feature is disabled, the shell-plugin path remains as fallback.
6. The `stratum` binary (`crates/kirkstratum-cli`) remains as a standalone CLI
   for non-KirkForge use cases.

## Gate

```
cargo fmt --check && cargo check --workspace --all-targets --features stratum &&
cargo clippy --all-targets --features stratum -- -D warnings &&
cargo test --locked --workspace --no-fail-fast --features stratum
```

Also run without the feature to confirm the fallback path:
```
cargo check --workspace --all-targets
cargo test --locked --workspace --no-fail-fast
```

## Done condition

### Shipped
- `kirkstratum-core` is linked into the binary when `--features stratum`.
- The 5 Stratum tools are direct Rust calls (no subprocess).
- The shell-plugin path still works when the feature is off.
- The 2 hooks are in-process Rust handlers (no shell scripts):
  - `StratumSessionStartHook` (session-start) — calls `kirkstratum_core` to emit the active compression ruleset via `tracing::info`.
  - `StratumPreToolBashHook` (pre-tool-bash) — validates stratum config via `kirkstratum_core`, fail-open.
  - Both registered under `#[cfg(feature = "stratum")]` in `src/session/executor/mod.rs`.
- ADR-046 pins the decision.
- Drift tests pass.
- Full gate green with and without the feature.
- Binary size impact measured and documented (expected: small — `kirkstratum-core` only adds `blake3` + `toml`, both already transitive deps).

### Deferred (honestly — see ADR-046)
- The `stratum_mode` config field (off/lite/full/ultra) is deferred (not needed for MVP — the pipeline runs in `full` mode by default).
