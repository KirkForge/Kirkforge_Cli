# Workorder 7.0 — Plugin System Consolidation

## Status: Planned

## Problem

After Workorders 6.6-6.9 fold the four satellite plugins (Stratum, Plugin3,
Draw, Video) into the core binary behind feature flags, the plugin system has
two parallel dispatch paths: (1) direct Rust calls for folded plugins and (2)
shell-script subprocesses for external plugins. The user-facing toggle surface
(`enabled_plugins`, `plugin_sources`, `/plugins` commands) must be unified so
that folded plugins and external plugins are managed through the same interface.

Additionally, the `kirkforge-plugin` self-plugin's 6 tools shell out to the Node
SDK. This is the one remaining shell-out in the default build. A decision is
needed: keep the Node SDK as an external dependency, or re-implement the
verification tools in Rust.

## Root cause

The fold-in (6.6-6.9) creates a two-path system that needs consolidation. The
Node SDK question predates the fold-in.

## Files to touch

- `src/session/plugin_tools/loader.rs` — unify the dispatch:
  - Folded plugins (feature-gated) register as direct Rust `Tool` impls.
  - External plugins (filesystem `kirkforge.toml`) continue to register as
    `PluginToolWrapper` shell-outs.
  - The `enabled_plugins` config controls both paths: a folded plugin name (e.g.
    "stratum") enables the Rust path; an external plugin name enables the shell
    path.
  - When a folded plugin's feature is off but the plugin is in `enabled_plugins`,
    the loader falls back to the shell-plugin path if the plugin dir exists
    (graceful degradation).
- `src/session/plugin_tools/mod.rs` — document the two-path dispatch.
- `src/shared/config/tools.rs` — the `default_plugin_sources` and
  `default_enabled_plugins` functions must account for which features are
  compiled in. Use `#[cfg(feature = "...")]` to conditionally include each
  folded plugin in the defaults.
- `src/session/plugin_tools/tests.rs` — update
  `default_plugin_sources_are_present_and_loadable` to handle feature-dependent
  defaults. Add tests for: folded plugin with feature on, folded plugin with
  feature off + shell fallback, external plugin.
- `src/tui/commands/plugins/mod.rs` — the `/plugins list` command should show
  whether each plugin is "compiled-in" or "external" and which feature gates it.
- `docs/adr/0050-plugin-system-consolidation.md` (new) — pin:
  - The two-path dispatch design (Rust for folded, shell for external).
  - The graceful-degradation rule (feature off + shell dir exists → shell path).
  - The Node SDK decision (keep as external or port to Rust — see below).
  - The unified toggle surface.
- `docs/adr/README.md` — add ADR-050.

## Approach

### Phase 1: Unify the dispatch

1. Refactor the loader so `load_workspace_plugins` checks whether a plugin name
   corresponds to a folded plugin (by `#[cfg(feature = "...")]`):
   - If the feature is on → register the Rust `Tool` impls + in-process hooks.
   - If the feature is off → check if the shell plugin dir exists → register
     `PluginToolWrapper` shell-outs (fallback).
   - If neither → skip with a warning.
2. The `enabled_plugins` list is the single source of truth for on/off. The
   `plugin_sources` map is only needed for external/shell plugins.

### Phase 2: Node SDK decision

3. Assess the 6 Node SDK tools (`plugin_verify`, `plugin_verify_workspace`,
   `plugin_audit_verify`, `plugin_doctor`, `plugin_health`, `plugin_tools`):
   - If they are thin wrappers around deterministic checks that could be Rust
     functions → port them to Rust and fold into core (no Node dependency).
   - If they depend on the Node ecosystem (ESLint, TypeScript, Ruff, Pyright) →
     keep as shell-outs with the Node SDK as an optional external dependency.
   - Document the decision in ADR-050.

### Phase 3: TUI and docs

4. `/plugins list` shows: plugin name, status (enabled/disabled), source
   (compiled-in/external), feature gate (if any).
5. Update ARCHITECTURE.md and README.md to reflect the consolidated system.

## Gate

```
cargo fmt --check && cargo check --workspace --all-targets --all-features &&
cargo clippy --all-targets --all-features -- -D warnings &&
cargo test --locked --workspace --no-fail-fast --all-features
```

Also default features:
```
cargo check --workspace --all-targets
cargo clippy --all-targets -- -D warnings
cargo test --locked --workspace --no-fail-fast
```

## Done condition

- Folded plugins register as direct Rust calls; external plugins use shell path.
- `enabled_plugins` is the single toggle for both paths.
- Graceful degradation works (feature off + shell dir → shell fallback).
- `/plugins list` shows source and feature gate.
- ADR-050 pins the design including the Node SDK decision.
- ARCHITECTURE.md and README.md updated.
- Drift tests pass.
- Full gate green with default features and `--all-features`.
- Depends on: Workorders 6.6-6.9 (all four folds must be done first).
